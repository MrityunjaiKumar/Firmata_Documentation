The feature registry defines allocated and proposed Firmata feature I\+Ds. The feature ID is the 2nd byte in the sysex message. An extended set of I\+Ds is also available by setting the initial ID byte to {\ttfamily 00H} and then following with a 2 byte ID. All bytes between {\ttfamily S\+T\+A\+R\+T\+\_\+\+S\+Y\+S\+EX} and {\ttfamily E\+N\+D\+\_\+\+S\+Y\+S\+EX} must have the most significant bit set to 0.

\hypertarget{md_protocol_feature-registry_autotoc_md25}{}\doxysection{Proposing a new feature}\label{md_protocol_feature-registry_autotoc_md25}
There are two different feature sets\+: \href{\#core-feature-set}{\texttt{ Core features}} and \href{\#optional-feature-set}{\texttt{ optional features}}. See the descriptions for each type of feature set below. To propose a new core feature, open an issue to start a discussion. To propose a new optional feature, \href{https://github.com/firmata/protocol/issues}{\texttt{ open an issue}}/and or a pull request adding a markdown file for the proposed feature. Also edit the \href{\#optional-feature-set}{\texttt{ optional feature set table}} to reserve an ID for the proposed feature and enter the status as \char`\"{}proposed\char`\"{}. If the proposed feature exposes a very specific device or device driver (a Neo\+Pixel light strip for example), assign an ID in the extended ID set ({\ttfamily 00H nnH nnH}).\hypertarget{md_protocol_feature-registry_autotoc_md26}{}\doxysection{Core feature set}\label{md_protocol_feature-registry_autotoc_md26}
Core features are related to functionality such as digital and analog I/O, querying information about the state and capabilities of the microcontroller board and the firmware running on the board. The core features are documented in the \href{https://github.com/firmata/protocol/blob/master/protocol.md}{\texttt{ protocol.\+md}} file and the full set of core features is versioned together using \href{http://semver.org/}{\texttt{ semver}} notation. The current protocol version is 2.\+5.\+1.

Firmata firmware should report the current protocol version (using the \href{https://github.com/firmata/protocol/blob/master/protocol.md\#message-types}{\texttt{ protocol version command\+: 0x\+F9}}) and implement the full set of current core features defined for that version (with the exception of very limited hardware which can implement a subset of the core feature set).

{\itshape The range 01H -\/ 0FH is reserved for user-\/defined features that are not added to this registry.}

$\ast$$\ast$7\+EH and 7FH are reserved because they have a special meaning to midi parsers.$\ast$\hypertarget{md_protocol_feature-registry_autotoc_md27}{}\doxysection{Optional feature set}\label{md_protocol_feature-registry_autotoc_md27}
Optional features extend the hardware capabilities beyond basic digital I/O and analog I/O (eg\+: I2C, Serial/\+U\+A\+RT, etc). Optional features also provide A\+P\+Is to interface with general components (eg\+: servo, stepper, rotary encoder, etc) as well as specific components (eg\+: D\+H\+T11, Neo\+Pixel, etc). The optional feature set also encompass functionality such as a general purpose scheduler A\+PI and a standardized device interface A\+PI. General features should use the single byte feature ID (allocating new I\+Ds in descending order). However, any feature that wraps a specific driver, specific sensor, one-\/off custom component, etc should use the extended feature ID ({\ttfamily 00H nnH nnH}) or should use the {\ttfamily D\+E\+V\+I\+C\+E\+\_\+\+Q\+U\+E\+R\+Y/\+R\+E\+S\+P\+O\+N\+SE} A\+PI.

Each feature should be documented in a markdown file and versioned independently using \href{http://semver.org/}{\texttt{ semver}} notation. In the case where a feature spans multiple I\+Ds (I2C for example), that entire set is documented in a single file and versioned together.
